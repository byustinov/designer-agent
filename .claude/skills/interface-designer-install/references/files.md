# Шаблоны файлов проекта

Читай этот файл на Шаге 2 блока 3. Записывай каждый файл через bash_tool с heredoc.
Подставляй реальные значения из ответов — не оставляй плейсхолдеры.

---

## CLAUDE.md

```bash
cat > "[Q11]/CLAUDE.md" << 'FILEEOF'
# Инструкции для Claude — [Q4]

## Контекст

[Q4] — [Q5]

## Правила поведения

- Никогда не создавай файлы (Figma, локальные, любые) без явного запроса
- Рабочая папка: [Q11] — сюда сохраняй все итоговые файлы
- Общайся по-русски
- Будь прямым: если что-то невозможно — говори сразу
- Не додумывай и не предполагай — задай уточняющий вопрос
- Если задача неоднозначна — уточни прежде чем приступать

## Агенты

- `.claude/agents/designer.md` — Figma, дизайн-система, UX-флоу, ревью, анализ

Правило маршрутизации: любая задача на Figma или дизайн — только через designer.
Оркестратор не рисует сам.

## Память

- `.claude/memory/domain/` — стабильные знания: контекст проекта, правила, договорённости
- `.claude/memory/history/` — лог сессий: решения, итоги, выводы

## Хук "пока"

Когда пользователь пишет "пока":
1. Спроси: «Что сохранить из этой сессии?»
2. Добавь строку в `.claude/memory/history/MEMORY.md`:
   `- [дата] — [краткое описание] — [ключевые решения]`
3. Если появились новые стабильные знания — обнови `.claude/memory/domain/project.md`
FILEEOF
```

```bash
# Domain Memory

- [Project Context](project.md) `[сегодняшняя дата]` — продукт, пользователи, задачи дизайнера

FILEEOF
```

## .claude/memory/domain/project.md

Подставляй все значения из ответов. Раздел «Правила» — только один, под тип процесса Q1.

```bash
cat > "[Q11]/.claude/memory/domain/project.md" << 'FILEEOF'
# [Q4]

## Продукт
[Q5]

## Пользователи
[Q6]

## Платформа
[Q7]

## Этап
[Q8]

## Задачи дизайнера
[Q9 — список через запятую]

## Дизайн-система
[Своя: ссылка на Figma-библиотеку]
[Готовая: название + ссылка на документацию]
[Нет: опираемся на платформенные гайдлайны]

## Тип рабочего процесса
[Q1]

## Правила

[Если продуктовый дизайн:]
- Любое изменение учитывает существующую дизайн-систему
- Задачи приходят из трекера — ссылаться на номер тикета
- Формат решения: проблема → гипотеза → решение → следующий шаг

[Если проектная разработка:]
- Перед решением — исследование и бенчмаркинг
- Структурировать по этапам: Discovery / Define / Design / Deliver
- Фиксировать принятые решения и причины отказа от альтернатив

[Если фриланс:]
- Опираться на ТЗ, явно указывать противоречия внутри него
- Разделять scope / out of scope
- Фиксировать договорённости с заказчиком отдельно от ТЗ

## Обновлено
[сегодняшняя дата]
FILEEOF
```
